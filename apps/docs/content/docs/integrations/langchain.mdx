LangChain is useful when you want provider abstraction, prompt composition, retrievers, or tool ecosystems while still using FitalyAgents as the runtime shell.

Runnable example: `examples/langchain-agent`

## Install

```bash
pnpm add @langchain/core @langchain/openai
```

Set your API key:

```bash
OPENAI_API_KEY=...
```

## LangChain as the LLM backend

The cleanest integration is to adapt a LangChain chat model to `IStreamingLLM`:

```ts
import { AIMessage, HumanMessage, SystemMessage, ToolMessage } from '@langchain/core/messages'
import { ChatOpenAI } from '@langchain/openai'
import type { IStreamingLLM, LLMStreamChunk } from 'fitalyagents'

class LangChainStreamingLLM implements IStreamingLLM {
  private model = new ChatOpenAI({
    model: process.env.LANGCHAIN_MODEL ?? 'gpt-4o-mini',
    temperature: 0.2,
  })

  async *stream(params: {
    system: string
    messages: Array<{ role: 'user' | 'assistant' | 'tool'; content: string; tool_call_id?: string }>
    tools?: Array<{ name: string; description?: string; input_schema?: unknown }>
  }): AsyncIterable<LLMStreamChunk> {
    const response = await this.model.invoke([
      new SystemMessage(params.system),
      ...params.messages.map((message) => {
        if (message.role === 'assistant') {
          return new AIMessage(message.content)
        }
        if (message.role === 'tool') {
          return new ToolMessage({
            content: message.content,
            tool_call_id: message.tool_call_id ?? 'tool_call_missing',
          })
        }
        return new HumanMessage(message.content)
      }),
    ])

    const text =
      typeof response.content === 'string'
        ? response.content
        : response.content
            .map((part) =>
              typeof part === 'string' ? part : 'text' in part ? String(part.text) : '',
            )
            .join('')
    if (text) {
      yield { type: 'text', text }
    }

    yield { type: 'end', stop_reason: 'end_turn' }
  }
}
```

Then use that adapter inside `InteractionAgent`.

## LangChain tools with `@fitalyagents/asynctools`

LangChain tools also fit naturally into `ToolRegistry`:

```ts
import { ToolRegistry, registerFunctionHandler } from '@fitalyagents/asynctools'
import { DynamicStructuredTool } from '@langchain/core/tools'
import { z } from 'zod'

const catalogTool = new DynamicStructuredTool({
  name: 'catalog_search',
  description: 'Search a product catalog',
  schema: z.object({ query: z.string() }),
  func: async ({ query }) => JSON.stringify({ results: [`Result for ${query}`] }),
})

const registry = new ToolRegistry()
registry.register({
  tool_id: 'catalog_search',
  executor: { type: 'ts_fn' },
  execution_mode: 'async',
})

registerFunctionHandler('catalog_search', async (input) => {
  return { result: await catalogTool.invoke(input) }
})
```

This lets LangChain handle tool implementation while FitalyAgents keeps concurrency, retries, circuit breaking, and governance outside the framework.

## Recommended split

- Use LangChain for prompts, model adapters, retrievers, and tool definitions
- Use FitalyAgents for event flow, session state, safety, approvals, and orchestration

That boundary keeps examples small and production behavior predictable.
